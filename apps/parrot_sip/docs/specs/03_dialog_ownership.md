# Dialog Discovery and Ownership Specification

**Version:** 1.0.0-draft
**Status:** APPROVED (documents existing implementation)
**Date:** 2025-12-05

## 1. Overview

This specification documents the **existing, correct** Dialog ownership pattern implemented in ParrotSip. This is NOT a new design - it describes the production-ready architecture already present in `dialog_statem.ex`.

**Key Principle:** Dialog processes create themselves and are discovered by entities via Registry-based lookup with deterministic IDs.

---

## 2. The Problem

**Question:** How does an Entity (UAS/UAC) find and communicate with its Dialog process?

**Wrong Approaches:**
- ❌ Entity creates Dialog → Race condition (Transaction also needs Dialog)
- ❌ Transaction creates Dialog and sends PID to Entity → Message passing complexity
- ❌ Global process registry with random IDs → No deterministic discovery

**Correct Approach (Existing Implementation):**
- ✅ Dialog creates itself
- ✅ Deterministic ID construction
- ✅ Registry-based discovery
- ✅ Entity registers as owner
- ✅ Dialog monitors Entity

---

## 3. Deterministic Dialog ID Construction

### 3.1 ID Format

```elixir
# UAS (Server) Dialog ID
"dialog:uas:#{call_id}:#{local_tag}:#{remote_tag}"

# UAC (Client) Dialog ID
"dialog:uac:#{call_id}:#{local_tag}:#{remote_tag}"
```

### 3.2 Example IDs

```elixir
# UAS receiving INVITE
call_id = "abc123@host.example.com"
local_tag = "server-tag-456"  # Generated by UAS in 200 OK
remote_tag = "client-tag-789"  # From INVITE From header

dialog_id = "dialog:uas:abc123@host.example.com:server-tag-456:client-tag-789"

# UAC sending INVITE
dialog_id = "dialog:uac:abc123@host.example.com:client-tag-789:server-tag-456"
```

### 3.3 Properties

**Deterministic:**
- Same inputs → same ID
- Both Dialog and Entity can compute same ID independently
- No coordination required

**Unique:**
- Call-ID + tags uniquely identify a dialog per RFC 3261
- Role prefix (uas/uac) prevents collision

**Immutable:**
- Dialog ID never changes for lifetime of dialog
- Re-INVITEs don't change tags

## 4. Dialog Self-Creation Pattern

### 4.1 When Dialog Creates Itself

**UAS Flow:**
```
1. TransactionStatem receives INVITE
2. Handler decides to answer (200 OK)
3. TransactionStatem sends 200 OK
4. TransactionStatem calls uas_maybe_create_dialog/2
5. Dialog.Supervisor.start_child({:uas, response, request})
6. Dialog process starts and registers with dialog_id
```

**UAC Flow:**
```
1. TransactionStatem sends INVITE
2. TransactionStatem receives 200 OK
3. TransactionStatem calls uac_maybe_create_dialog/2
4. Dialog.Supervisor.start_child({:uac, response, request})
5. Dialog process starts and registers with dialog_id
```

### 4.2 Existing Implementation

**From `dialog_statem.ex:792-799`:**

```elixir
defp uas_maybe_create_dialog(%Message{} = resp_sip_msg, %Message{} = req_sip_msg) do
  if should_create_dialog?(resp_sip_msg, req_sip_msg) do
    case ParrotSip.Dialog.Supervisor.start_child({:uas, resp_sip_msg, req_sip_msg}) do
      {:ok, pid} ->
        Logger.info("Dialog created successfully with PID: #{inspect(pid)}")
        resp_sip_msg
      {:error, reason} ->
        Logger.error("Failed to create dialog: #{inspect(reason)}")
        resp_sip_msg
    end
  end
end
```

**Key Points:**
- Called from Transaction layer
- Happens during 200 OK processing
- Dialog exists BEFORE Entity needs it (no race condition)
- Uses DynamicSupervisor for dynamic child creation

### 4.3 Dialog Registration

When Dialog starts, it immediately registers:

```elixir
def init({role, resp_msg, req_msg}) do
  dialog_id = build_dialog_id(role, call_id, local_tag, remote_tag)

  # Register in global Registry
  {:ok, _} = Registry.register(ParrotSip.Registry, dialog_id, %{})

  # Dialog is now discoverable
  data = %{
    id: dialog_id,
    role: role,
    owner: nil,  # Will be set later via set_owner/2
    # ... other fields
  }

  {:ok, :early, data}
end
```

## 5. Entity Discovery of Dialog

### 5.1 When Entity Discovers Dialog

**UAS Timeline:**
```
1. UAS process created from INVITE
2. UAS transitions :incoming → :ringing (sends 180)
3. Handler decides to answer
4. UAS transitions :ringing → :answering (sends 200 OK)
5. UAS :answering state :enter callback fires
6. UAS constructs dialog_id
7. UAS calls Dialog.uas_find(dialog_id)
8. UAS calls Dialog.set_owner(dialog_pid, dialog_id)
```

**UAC Timeline:**
```
1. UAC process created by application
2. UAC sends INVITE
3. UAC receives 200 OK
4. UAC transitions :calling → :answered
5. UAC sends ACK
6. UAC transitions :answered → :established
7. UAC :established state :enter callback fires
8. UAC constructs dialog_id
9. UAC calls Dialog.uac_find(dialog_id)
10. UAC calls Dialog.set_owner(dialog_pid, dialog_id)
```

### 5.2 Existing Implementation

**From `dialog_statem.ex:286-297`:**

```elixir
@spec uas_find(dialog_id :: String.t()) :: {:ok, pid()} | {:error, :not_found}
def uas_find(dialog_id) do
  case Registry.lookup(ParrotSip.Registry, dialog_id) do
    [{pid, _}] -> {:ok, pid}
    [] -> {:error, :not_found}
  end
end

@spec uac_find(dialog_id :: String.t()) :: {:ok, pid()} | {:error, :not_found}
def uac_find(dialog_id) do
  case Registry.lookup(ParrotSip.Registry, dialog_id) do
    [{pid, _}] -> {:ok, pid}
    [] -> {:error, :not_found}
  end
end
```

### 5.3 Usage in UAS State Machine

**In UAS gen_statem:**

```elixir
# State: :answering
# Event: {:enter, :answering}
def handle_event(:enter, _old_state, :answering, data) do
  # Construct dialog ID from INVITE
  dialog_id = build_uas_dialog_id(data.invite)

  case Dialog.uas_find(dialog_id) do
    {:ok, dialog_pid} ->
      # Dialog found - register as owner
      :ok = Dialog.set_owner(dialog_pid, dialog_id)

      # Store dialog PID
      data = %{data | dialog: dialog_pid, dialog_id: dialog_id}
      {:keep_state, data}

    {:error, :not_found} ->
      # Dialog not created yet (race condition)
      # Retry after brief delay
      Process.send_after(self(), :retry_find_dialog, 100)
      {:keep_state, data}
  end
end

def handle_event(:info, :retry_find_dialog, :answering, data) do
  # Retry logic
  handle_event(:enter, :answering, :answering, data)
end
```

### 5.4 Why Retry May Be Needed

**Timing:**
- Dialog creation happens asynchronously in Transaction layer
- Entity transition to :answering happens in Entity process
- Small window where Entity reaches :answering before Dialog registered

**Solution:**
- Retry with 100ms delay (more than enough for Dialog to register)
- In practice, Dialog is usually ready on first attempt

## 6. Ownership Registration

### 6.1 The set_owner/2 API

**From `dialog_statem.ex:501-510`:**

```elixir
@spec set_owner(pid(), String.t()) :: :ok
def set_owner(pid, dialog_id) when is_pid(pid) do
  case Registry.lookup(ParrotSip.Registry, dialog_id) do
    [{dialog_pid, _}] ->
      :gen_statem.cast(dialog_pid, {:set_owner, pid})
    [] ->
      :ok
  end
end
```

**What it does:**
1. Looks up Dialog process in Registry
2. Sends `{:set_owner, entity_pid}` cast to Dialog
3. Returns `:ok` immediately (async operation)

### 6.2 Dialog Processes set_owner Cast

**From `dialog_statem.ex:656-667`:**

```elixir
def handle_event(:cast, {:set_owner, owner_pid}, _state, data) do
  data = process_set_owner(data, owner_pid)
  {:keep_state, data}
end

defp process_set_owner(data, pid) do
  # Monitor the owner
  ref = Process.monitor(pid)

  # Store owner and monitor ref
  data
  |> Map.put(:owner, pid)
  |> Map.put(:owner_ref, ref)
end
```

**Effects:**
1. Dialog stores `owner: entity_pid`
2. Dialog monitors Entity via `Process.monitor(entity_pid)`
3. Dialog now knows where to send events
4. Dialog detects if Entity crashes

## 7. Event Propagation

### 7.1 Event Flow Direction

```
Transaction Layer (Layer 2)
    ↓ sends {:tx_event, ...}
Dialog Layer (Layer 2)
    ↓ sends {:dialog_event, ...}
Entity Layer (Layer 3: UAS/UAC)
    ↓ sends {:uas_event, ...} or {:uac_event, ...}
Application Layer (Layer 4: Session/Handler)
```

### 7.2 Example: ACK Received

**Full Flow:**

```
1. TransactionStatem receives ACK from network
   └─> State: waiting_for_ack → confirmed

2. TransactionStatem sends to Dialog (owner)
   └─> send(dialog_pid, {:tx_event, :ack_received})

3. Dialog processes ACK
   └─> Updates internal state
   └─> State: early → confirmed

4. Dialog sends to Entity (owner)
   └─> send(entity_pid, {:dialog_event, :ack_received})

5. UAS processes ACK
   └─> State: answering → established
   └─> Notifies application: {:uas_established, uas_pid}

6. B2BUA.Session processes UAS established
   └─> Checks if both legs ready
   └─> Session state: connecting → established
```

### 7.3 Example: Timer H Timeout

**Full Flow:**

```
1. TransactionStatem Timer H fires (32s, no ACK)
   └─> State: proceeding → terminated

2. TransactionStatem sends to Dialog
   └─> send(dialog_pid, {:tx_event, :timer_h_timeout})

3. Dialog processes timeout
   └─> State: early → terminated

4. Dialog sends to Entity
   └─> send(entity_pid, {:dialog_event, :timer_h_timeout})

5. UAS processes timeout
   └─> State: answering → terminated
   └─> Notifies application: {:uas_timeout, uas_pid}

6. B2BUA.Session handles timeout
   └─> Terminates session
```

### 7.4 Event Types

**Transaction → Dialog:**
- `{:tx_event, :ack_received}`
- `{:tx_event, :timer_h_timeout}`
- `{:tx_event, {:request_received, request}}`

**Dialog → Entity:**
- `{:dialog_event, :ack_received}`
- `{:dialog_event, :timer_h_timeout}`
- `{:dialog_event, {:bye_received, bye_msg}}`
- `{:dialog_event, {:reinvite, invite_msg}}`

**Entity → Application:**
- `{:uas_established, uas_pid}`
- `{:uas_timeout, uas_pid}`
- `{:uas_bye, uas_pid, bye_msg}`
- `{:uac_answered, uac_pid, sdp}`

## 8. Crash Recovery

### 8.1 Entity Crashes

**Scenario:**
- UAS/UAC process crashes (bug, timeout, OOM, etc.)
- Dialog is still alive

**What Happens:**

```elixir
# In Dialog gen_statem
def handle_info({:DOWN, ref, :process, pid, reason}, state, data) do
  if ref == data.owner_ref do
    Logger.warn("Dialog owner crashed: #{inspect(reason)}")

    # Clean up and terminate dialog
    {:stop, :normal, data}
  else
    {:keep_state, data}
  end
end
```

**Result:**
- Dialog detects crash via monitor
- Dialog terminates (let-it-crash philosophy)
- Call ends gracefully

### 8.2 Dialog Crashes

**Scenario:**
- Dialog process crashes
- Entity is still alive

**What Happens:**

```elixir
# In UAS/UAC gen_statem
def handle_info({:DOWN, ref, :process, pid, reason}, state, data) do
  if ref == data.dialog_ref do
    Logger.error("Dialog crashed: #{inspect(reason)}")

    # Can't continue without dialog - terminate entity
    {:stop, :normal, data}
  else
    {:keep_state, data}
  end
end
```

**Result:**
- Entity detects crash via monitor (if Entity monitors Dialog)
- Entity terminates
- Call ends

**Alternative:**
- Entity could attempt to recreate Dialog
- In practice: let-it-crash is simpler and correct

### 8.3 Both Crash

**Scenario:**
- Session crashes, taking both UAS and UAC with it

**What Happens:**
- Dialogs detect Entity crashes via monitors
- Dialogs terminate
- Transaction layer cleans up
- Supervision tree handles restart (if configured)

**OTP Benefits:**
- No orphaned processes
- No resource leaks
- Clean crash recovery

## 10. Registry Usage

### 10.1 Registry Configuration

```elixir
# In application.ex supervisor tree
{Registry, keys: :unique, name: ParrotSip.Registry}
```

**Properties:**
- `:unique` - Each dialog_id can only have one process
- Prevents accidental duplicate dialogs
- Fast O(1) lookups

### 10.2 Registration

```elixir
# In Dialog init/1
{:ok, _} = Registry.register(ParrotSip.Registry, dialog_id, %{})
```

**Metadata:**
- Currently empty `%{}`
- Could store dialog role, state, etc. for debugging

### 10.3 Lookup

```elixir
# From Entity
case Registry.lookup(ParrotSip.Registry, dialog_id) do
  [{pid, _metadata}] -> {:ok, pid}
  [] -> {:error, :not_found}
end
```

### 10.4 Automatic Cleanup

When Dialog terminates:
- Registry automatically removes entry
- No manual cleanup needed
- Prevents stale registrations

## 11. Why This Design Works

### 11.1 No Race Conditions

**Problem:** What if Entity looks for Dialog before it's created?

**Solution:**
1. Dialog creates itself when Transaction sends 200 OK
2. Entity looks for Dialog in :answering/:established state
3. :answering/:established happens AFTER 200 OK sent
4. Therefore: Dialog always exists before Entity looks

**Fallback:**
- Retry logic handles edge cases
- 100ms delay more than enough

### 11.2 Deterministic Discovery

**Problem:** How do two processes agree on an ID without coordination?

**Solution:**
- ID derived from SIP message fields (Call-ID, tags)
- Both processes have same SIP message
- Both compute same ID
- No message passing needed

### 11.3 Clean Ownership

**Problem:** What if Entity can't send to Dialog?

**Solution:**
- Entity stores `dialog: pid()`
- Entity can send directly: `send(dialog_pid, msg)`
- No registry lookup needed for sending
- Registry only used for discovery

### 11.4 Crash Safety

**Problem:** What if Dialog or Entity crashes?

**Solution:**
- Process monitoring detects crashes
- Mutual monitoring: Dialog monitors Entity, Entity monitors Dialog
- Both terminate together
- No orphaned processes

### 11.5 OTP Compliance

**Follows OTP principles:**
- ✅ Processes are isolated
- ✅ Let-it-crash philosophy
- ✅ Supervision trees handle restarts
- ✅ No shared state (only messages)
- ✅ Deterministic behavior

## 12. Implementation Checklist

When implementing Entity (UAS/UAC):

- [ ] Construct dialog_id from INVITE in appropriate state
- [ ] Call `Dialog.uas_find(dialog_id)` or `Dialog.uac_find(dialog_id)`
- [ ] Handle `{:error, :not_found}` with retry logic
- [ ] Call `Dialog.set_owner(dialog_pid, dialog_id)` when found
- [ ] Store `dialog_pid` in entity data
- [ ] Monitor dialog: `Process.monitor(dialog_pid)`
- [ ] Handle `{:DOWN, ref, :process, dialog_pid, reason}`
- [ ] Handle `{:dialog_event, event}` messages
- [ ] Forward events to application owner

When implementing Dialog features:

- [ ] Construct dialog_id in `init/1`
- [ ] Register in Registry immediately
- [ ] Handle `{:set_owner, entity_pid}` cast
- [ ] Monitor entity: `Process.monitor(entity_pid)`
- [ ] Store `owner: entity_pid` in data
- [ ] Send events to owner: `send(owner_pid, {:dialog_event, event})`
- [ ] Handle `{:DOWN, ref, :process, owner_pid, reason}`
- [ ] Unregister from Registry on termination (automatic)

## 13. Testing Strategy

### 13.1 Unit Tests

**Test Dialog Self-Creation:**
```elixir
test "dialog creates itself when transaction sends 200 OK" do
  # Simulate transaction sending 200 OK
  resp = build_200_ok_response()
  req = build_invite_request()

  # Dialog should be created
  {:ok, dialog_pid} = Dialog.Supervisor.start_child({:uas, resp, req})

  # Dialog should be registered
  dialog_id = build_dialog_id(:uas, req.call_id, resp.to_tag, req.from_tag)
  assert [{^dialog_pid, _}] = Registry.lookup(ParrotSip.Registry, dialog_id)
end
```

**Test Entity Discovery:**
```elixir
test "entity discovers dialog via registry" do
  # Create dialog first
  dialog_id = "dialog:uas:call123:local:remote"
  {:ok, dialog_pid} = start_test_dialog(dialog_id)

  # Entity looks up dialog
  assert {:ok, ^dialog_pid} = Dialog.uas_find(dialog_id)
end
```

**Test Ownership Registration:**
```elixir
test "entity registers as owner" do
  {:ok, dialog_pid} = start_test_dialog()
  {:ok, entity_pid} = start_test_entity()

  # Entity registers as owner
  :ok = Dialog.set_owner(dialog_pid, dialog_id)

  # Dialog monitors entity
  dialog_state = :sys.get_state(dialog_pid)
  assert dialog_state.owner == entity_pid
  assert is_reference(dialog_state.owner_ref)
end
```

### 13.2 Integration Tests

**Test Full UAS Flow:**
```elixir
test "complete UAS call with dialog ownership" do
  # Send INVITE
  {:ok, tx_pid} = send_invite()

  # Create UAS entity
  {:ok, uas_pid} = UAS.start_link(invite: invite, ...)

  assert_receive {:uas_answered, ^uas_pid}

  # Send ACK
  send_ack()

  # UAS should receive ACK via dialog
  assert_receive {:uas_established, ^uas_pid}

```

### 13.3 Crash Recovery Tests

**Test Entity Crash:**
```elixir

  # Dialog should terminate

end
```
